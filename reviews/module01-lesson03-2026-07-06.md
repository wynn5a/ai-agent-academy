## Findings (most severe first)

### [Blocker → fixed] "Complete working" and "production" loops break under Sonnet 5's own default thinking behavior
- **Where:** `content/modules/module01/lessons/03-tool-calling.ts:70` and `:259`
- **Standard:** 1. Factual accuracy (internal consistency) / 3. Comprehensive depth
- **Issue:** Both the lesson's flagship example ("complete working tool loop (Anthropic, raw SDK)") and its hardened "error-returning executor + guarded loop" call `model="claude-sonnet-5"` without setting `thinking`, then do `print(resp.content[0].text)` / `return resp.content[0].text` on the non-tool-use branch. Lesson 2 of this same module states explicitly: *"leave `thinking` out of the request entirely and Sonnet 5 runs adaptive automatically"* — meaning a `thinking` content block lands **before** the `text` block in `resp.content` whenever the model reasons before answering, which adaptive thinking will do for a nontrivial fraction of real requests. When that happens, `resp.content[0]` is a thinking block (no `.text` attribute matching the text block), so this "complete" code throws or silently prints the wrong thing.
- **Why it matters:** This is exactly the kind of forward-reference/happy-path bug the Academy's own quiz-drill instincts are trying to train out of learners — and finding it in the flagship code sample undercuts the lesson's credibility on the "the model never executes anything, your code does the work" framing it opens with. A learner who copies this pattern into their own agent will hit a confusing runtime error the first time thinking kicks in, exactly the "silent agent failure" class of bug Lesson 2 warns about.
- **Suggested fix:** Replace both call sites with a block-type scan consistent with Lesson 2's own guidance, e.g. `next(b.text for b in resp.content if b.type == "text")`, or explicitly set `thinking` to `None`/omit adaptive reasoning for this teaching example and say why. Either way, note inline that indexing `content[0]` is unsafe once thinking is in play.

### [Major → fixed] No comparison table for what is fundamentally a field-name mapping problem
- **Where:** whole lesson; most concentrated across `:36-216`
- **Standard:** 6. Effective tables, diagrams & animations
- **Issue:** The lesson's core teaching content — Anthropic vs. OpenAI field names (`input_schema`/`parameters`, `tool_use`/`function_call`, `tool_result`/`function_call_output`, `tool_use_id`/`call_id`, `is_error`/no equivalent, parsed dict vs. JSON string arguments) — is exactly the kind of multi-way comparison Lesson 2 handles with a table (the `stop_reason`/`finish_reason`/`status` tables). Here it's scattered across two near-duplicate full code walkthroughs (`:36-142` and `:144-216`, one loop-shaped and one single-shot, both showing both providers via tabs/variants) plus prose asides in several `explanation` fields. A reader has to reconstruct the mapping themselves from two long code blocks instead of glancing at one table.
- **Why it matters:** This is precisely the standard's "smell": a multi-way comparison done in code/prose that begs for a table. It also means the second code block (`:144-216`) mostly re-teaches the same shape difference the first block's `variants` already covered, padding the lesson without adding new information — at the cost of lesson budget that could go to something not otherwise shown, like a code sample for strict mode (see next finding).
- **Suggested fix:** Collapse the two near-duplicate walkthroughs into one (keep the loop-shaped one, since only it demonstrates the full multi-turn mechanic), and add a `table` section mapping the two providers' field names side by side, the way Lesson 2 does for stop signals.

### [Minor → fixed] Strict mode is described in prose only, with no code example
- **Where:** `content/modules/module01/lessons/03-tool-calling.ts:219-224`
- **Standard:** 3. Comprehensive depth / 5. Engaging interactivity
- **Issue:** The strict-mode paragraph names exact field names and constraints (`"strict": True` top-level, `additionalProperties: False`, every property in `required`) with the same precision as every other code-backed claim in this lesson and Lesson 2 — but it's the only technical mechanism in the lesson never shown in a code block. Verified against current Anthropic docs: the paragraph's claims are accurate (top-level `strict: true`, `additionalProperties: false`, `required` listing every property), so this is a presentation gap, not a factual one.
- **Why it matters:** The lesson calls this feature something to "use... for every tool whose arguments feed real side effects" — that's a production-critical recommendation, and the exemplar pattern in this Academy is to always back such recommendations with runnable code the reader can adapt.
- **Suggested fix:** Add a short code block showing one `TOOLS` entry with `"strict": True` and a schema with `additionalProperties: False`, ideally reusing the `get_weather` tool already established. Freed-up space from collapsing the earlier duplicate walkthrough (previous finding) covers this without growing the lesson.

### [Minor → fixed] Unqualified `stop_reason` mention in a bullet that's otherwise carefully split by provider
- **Where:** `content/modules/module01/lessons/03-tool-calling.ts:30`
- **Standard:** 2. Clarity & fluency
- **Issue:** `'**Model decides** it needs a tool: the response contains a `tool_use` block (Anthropic) / `tool_calls` array (OpenAI)... `stop_reason` is `"tool_use"`.'` The bullet is careful to tag the block/array naming per provider, then drops that discipline for the trailing `stop_reason` clause, which is Anthropic-only terminology (Lesson 2 establishes OpenAI uses `finish_reason`/`status` instead). A reader skimming this bullet in isolation could read `stop_reason` as universal.
- **Why it matters:** Small, but this is the kind of imprecision interviewers probe for specifically ("which field, on which API?") — the lesson's own "Interview angle" callout later says exactly that.
- **Suggested fix:** Append `(finish_reason: "tool_calls" / status stays "completed" with a function_call output item, on OpenAI)` or similar, mirroring how carefully Lesson 2's keypoints handle this split.
