The one finding worth blocking a "provider-fluent" claim on is the total absence of the provider-native multi-agent frameworks (Major #1): a module literally titled "Multi-Agent Systems & Frameworks" that teaches "handoffs" as a concept but never mentions that `handoffs` is the OpenAI Agents SDK's flagship primitive leaves a learner exposed in an OpenAI-shop interview. ## Findings (most severe first)

### [Major] The two provider-native multi-agent SDKs are never named — the module is single-track LangGraph
- **Where:** whole module; acutely `lessons/04-orchestrators-workers-and-handoffs.ts` (the entire lesson teaches "handoffs" and "orchestrator-workers" generically) and `lessons/01-what-frameworks-actually-buy-you.ts:62` (names LangChain/LangGraph/AutoGen/CrewAI as the frameworks that "show up by name" — omits both provider SDKs).
- **Standard:** 4 (Anthropic + OpenAI) and 7 (role alignment).
- **Issue:** The OpenAI Agents SDK's flagship primitives are literally `handoffs` and "agents-as-tools" (manager pattern) — precisely the two shapes Lesson 4 teaches. It's the single most prominent named implementation of exactly this concept, and the module never mentions it. Anthropic's side (code-orchestrated subagents / the Claude Agent SDK, orchestrator-worker) is referenced only obliquely via one resource link. There's a concrete, teachable difference the module misses entirely: OpenAI SDK handoffs transfer the **full conversation history by default** (with optional structured metadata) — which is *exactly* the full-history anti-pattern Lesson 4 spends a section arguing against — whereas LangGraph uses shared checkpointed state and Anthropic's subagents get scoped briefs. That contrast is the "where they differ" gold Standard 4 exists to surface.
- **Why it matters:** Postings name both providers (research doc, "Most Requested Technical Skills"). A senior candidate asked "how do handoffs work in the OpenAI Agents SDK?" or "how does that compare to LangGraph's shared state?" who only knows the generic concept reads as someone who learned multi-agent from one framework's docs. Knowing the provider-native primitives and their default behaviors is the fluency interviewers probe.
- **Suggested fix:** Add a short comparison table or `tab-group` (in L1 or the top of L4) mapping concept → provider-native primitive: LangGraph = shared checkpointed `StateGraph`; OpenAI Agents SDK = `handoffs` (peer transfer, full-history default) + agents-as-tools (manager); Anthropic = orchestrator + subagents (Claude Agent SDK). Then one sentence in L4's handoff-payload section: note that OpenAI's `handoffs` defaults to transferring the whole history — the anti-pattern this lesson warns against — and you opt into a structured brief via handoff metadata. This turns the module's biggest gap into its sharpest both-provider teaching moment, and independently, OpenAI's own "start with one agent, split only for isolation/parallelism/policy" guidance corroborates Lesson 5's thesis — worth citing.

### [Minor] Sonnet 5 sampling-parameter claim is over-broad vs. the exemplar's careful scoping
- **Where:** `lessons/05-when-multi-agent-is-worth-it.ts:125` (LLM-as-judge code explanation): "current frontier Claude models reject `temperature`/`top_p`/`top_k` entirely (Module 1)."
- **Standard:** 1 (factual accuracy / internal consistency).
- **Issue:** The exemplar (Module 1, Lesson 2) is deliberately precise about this exact distinction: Opus 4.7/4.8 reject those params **outright**, but Sonnet 5 rejects only **non-default** values (omitting or passing the default is accepted). The judge code here uses `claude-sonnet-5`, so "reject entirely" contradicts the exemplar's own scoping for the very model being used. The practical conclusion (you can't pin the judge to temperature 0 for consistency) is still correct — temp 0 is a non-default value — so this is a precision/consistency slip, not a broken claim.
- **Why it matters:** The Academy's credibility rests on the version-scoped precision the exemplar models; a learner who repeats "Claude rejects temperature entirely" in an interview loses the nuance that separates "has read the docs" from "has shipped against them."
- **Suggested fix:** "current frontier Claude models reject non-default `temperature`/`top_p`/`top_k` (Opus outright; Sonnet 5 accepts only its default) — so pinning the judge to temperature 0 for consistency isn't available (Module 1)."

### [Minor] The core new structural concept (dynamic fan-out) is taught without a diagram
- **Where:** `lessons/02-state-nodes-and-edges.ts:118-120` (dynamic fan-out / map-reduce paragraph and the fan-out row of the edge table) — no `animation` in the lesson.
- **Standard:** 6 (tables/diagrams/animations).
- **Issue:** Fan-out → parallel workers → reduce-into-shared-state is the module's signature new mechanism and a temporal/pipeline concept a diagram would collapse. The `multi-agent` animation that visualizes exactly this (planner fans out to parallel searchers) exists but is used **only in lab.ts**; L4's `workflow-patterns` shows the three pattern *shapes* but not the map→reduce mechanics. L2, where the concept is actually taught, has no visual.
- **Why it matters:** This is the one place in the module where prose is doing work a picture does better; a reader forms the fan-out/reduce mental model here, and it's the backbone of the lab.
- **Suggested fix:** Reuse the `multi-agent` animation (a valid `AnimationName`) at the fan-out paragraph in L2, with a caption scoped to map→reduce. (L3's checkpoint/resume/time-travel is also a temporal mechanism with no diagram, but there's no obviously-fitting existing `AnimationName`, so that's a flagged absence, not an easy fix.)

### [Minor] A2A protocol is gestured at but never named
- **Where:** `lessons/04-orchestrators-workers-and-handoffs.ts:146` (the "message passing (actor-style)" / "agent-to-agent messaging model" discussion).

- **Suggested fix:** One clause in the message-passing paragraph: "…frameworks built around an actor or agent-to-agent messaging model (the emerging **A2A** protocol formalizes exactly this cross-org boundary; Module 6)…"
