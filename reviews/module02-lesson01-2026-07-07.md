## Findings (most severe first)

### [Major] OpenAI Responses API parameter is `max_output_tokens`, not `max_tokens`
- **Where:** `content/modules/module02/lessons/01-the-loop-that-makes-an-agent.ts:180`
- **Standard:** 1. Factual accuracy / 4. Anthropic + OpenAI compatibility
- **Issue:** The OpenAI code variant's explanation says: *"...and `max_tokens` is optional where Anthropic requires it."* The Responses API (`client.responses.create`, which is what the code block actually calls) has no `max_tokens` parameter at all — the field is `max_output_tokens`. Confirmed against the `openai-python` SDK signature: `max_output_tokens: Optional[int] | Omit = omit` on `responses.create()`. Passing `max_tokens=...` to this method would be an unexpected keyword, not an optional/ignored one.
- **Why it matters:** This is exactly the kind of provider-difference claim interviewers probe (standard 4 exists for this reason), and it's concrete enough that a learner could carry the wrong field name into code or a whiteboard session. Notably, the course itself states the correct name elsewhere — `module01/lessons/02-sampling-and-streaming.ts:238-239` correctly uses `max_output_tokens` twice in the Responses-API `status` table — so this isn't a case of an unresolved API ambiguity, it's an internal inconsistency.
- **Not isolated to this file:** The identical wrong phrasing ("`max_tokens` is optional ... where Anthropic requires it") also appears in `module01/lessons/01-messages-are-the-only-state.ts:105` and `module02/lessons/03-workflows-vs-agents.ts:127`. Worth a single sweep across all three rather than fixing this file alone.
- **Suggested fix:** Replace `max_tokens` with `max_output_tokens` is optional (the Responses API applies a model default when omitted) where Anthropic's `max_tokens` is required on every call.
