## Findings (most severe first)

### [Minor] Historical ReAct code block is Anthropic-specific but untagged and unpaired
- **Where:** `content/modules/module02/lessons/02-react-and-planning.ts:21-57`
- **Standard:** 4. Anthropic + OpenAI compatibility
- **Issue:** The `react_step` code block calls `client.messages.create(...)` with `stop_sequences=["Observation:"]` — that's Anthropic SDK syntax, not provider-neutral pseudocode. It carries no `provider: "claude"` field and has no `variants` entry, so it defaults to rendering as "neutral" (per `lib/types.ts`, default `provider` is `"neutral"`). Every other lesson in this module (01, 03, 04, 05) tags every SDK-calling code block this way and pairs it with an OpenAI variant; this is the one exception. The `language: "text"` softmax-style blocks elsewhere are correctly left untagged because they're genuinely provider-neutral — this block isn't in that category, it's real SDK code.
- **Why it matters:** A learner skimming tabs would read this as "the" way to do prompt-based ReAct without realizing it's Anthropic-flavored (`stop_sequences` vs OpenAI's `stop`), and misses the easy opportunity to show the OpenAI-side equivalent of the same historically-important trick (`stop=["Observation:"]` on Chat Completions, or `client.responses.create(..., messages=..., )` — Responses API stop handling differs subtly and would itself be a fact worth teaching).
- **Suggested fix:** Add `provider: "claude"` to this block and a short OpenAI `variants` entry showing the equivalent `stop=[...]` on Chat Completions, with an explanation noting any difference (e.g. Chat Completions caps stop sequences at 4, per lesson 2 of module 1). If the intent is truly to keep this pseudocode-neutral, drop the SDK-specific method name/kwargs instead.
